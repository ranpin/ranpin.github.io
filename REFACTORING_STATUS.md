### ⚠️ 遇到的问题

在尝试将 App.jsx 迁移到使用 Zustand store 时，发现以下问题：

1. **函数定义缺失**：App.jsx 中调用了多个事件处理函数（如 `handleDeleteWithUndo`、`handleInsertAt`、`handleInlineSave` 等），但这些函数的定义位置不明确。

2. **构建成功但搜索不到定义**：虽然 `grep` 搜索找不到这些函数的 `const` 或 `function` 定义，但 Webpack 构建成功，说明它们确实存在。可能是：
   - 函数定义在文件很后面的位置（1973行文件中）
   - 使用了不同的定义语法
   - 通过闭包或其他方式定义

3. **保守策略必要性**：由于无法准确定位所有依赖关系，直接替换状态定义可能导致运行时错误。

## 🎯 调整后的策略

### 方案 A：渐进式迁移（推荐）

**步骤**：
1. **保留原有 useState**，同时引入 Zustand store
2. **逐步将 setState 调用替换为 store actions**
3. **每次替换后测试功能**
4. **确认无误后再删除旧的 useState**

**优点**：
- 风险低，可随时回退
- 每步都可验证
- 不会破坏现有功能

**缺点**：
- 过渡期代码略显冗余
- 需要更多测试

---

### 方案 B：完整重写 App.jsx

**步骤**：
1. 从 git 历史中提取所有事件处理函数
2. 创建新的 App.jsx，使用 Zustand store
3. 重新实现所有事件处理逻辑

**优点**：
- 代码干净，无遗留
- 架构清晰

**缺点**：
- 工作量大
- 容易遗漏细节
- 测试成本高

---

## 💡 建议

**采用方案 A（渐进式迁移）**，原因：
1. 用户要求"不允许明显改变当前的网站布局和已有的功能"
2. 渐进式迁移最符合这一要求
3. 可以边迁移边测试，确保功能完整性

## 🔍 技术发现

### 关于缺失函数的调查结果

通过深入分析，发现这些"缺失"的函数实际上**可能并不存在于原始代码中**，而是：

1. **内联箭头函数**：直接在 JSX 中定义的匿名函数
2. **未实现的占位符**：代码中有调用但未实现（会导致运行时错误）
3. **动态生成**：通过高阶函数或工厂函数生成

需要进一步检查原始代码的实际运行情况来确认。

## 📝 经验教训

1. **大规模重构需谨慎**：1973行的单文件不适合一次性重写
2. **先理解再修改**：应先用工具（如 AST 分析器）理清依赖关系
3. **小步快跑**：每次只改一小部分，立即测试
4. **保留回退路径**：Git 提交点很重要
